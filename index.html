学习萨克斯-准备篇

Veröffentlicht am 2019-09-01 | in music

为什么学萨克斯

萨克斯音色优美，高端优雅，吹奏需要一定的肺活量和气力。个人比较喜欢此类乐器，学习演奏萨克斯还能锻炼身体，因为萨克斯管自身重量不轻。

本人从小就喜欢听萨克斯音乐，只是小时候条件有限，没有如愿学习。工作了一段时间以后稳定下来才有时间来思考，学习一门乐器。有次偶然听了Kenny G的《Sentimental》，心中的念头又强烈了起来。经过了反复犹豫，纠结，权衡以后。。。。。。还是决定要行动起来，试一试。

如何选择萨克斯

萨克斯有几个主流类型，高音萨克斯(降b)，中音萨克斯(降e)，次中音(降b)，低音萨克斯。大家最常听的萨克斯曲《回家》由Kenny G大师使用高音萨克斯演奏。在网上了解了这几种萨克斯的特点，对于新手来说，中音和次中音是最适合学习的，指法上好协调。高音和低音不很适合（大神可以忽略）。所以我也就采纳了过来人的建议，决定从中音萨克斯入手（对爵士乐没要求，次中音就没考虑了。。其实是我比较懒，网上看到的第一篇资料就是中音的）。

上面的萨克斯样子参考图： 依次为高音，中音萨克斯。

那么重点来了，作为初学者，选什么牌子的萨克斯好？

网上逛了一圈，五花八门，但是可以简单分为以下几大类：

  • 欧美品牌：塞尔玛等
  • 日本品牌：柳泽，雅马哈等
  • 台湾品牌：杰普特，工匠等

本人对雅马哈（YAMAHA）比较熟悉，雅马哈做工皮实，经得起折腾，适合我这种不太爱惜乐器的同学。另外雅马哈的音色对于初学者来说也还可以。在塞尔玛序言，杰普特，柳泽，雅马哈之间考虑了一段时间，最后选择了雅马哈200dr的中音管子，笛头和哨片都使用原厂包装，没有另购。管子是从一位博主那里采购，有和雅马哈线下琴行合作，价钱合适，比官方价便宜1300大洋。 如果你壕，可以考虑雅马哈yas62，或者柳泽，塞尔玛甚至更高端的品牌型号，它们的音色更美。

萨克斯的发声原理

买回来以后，迫不及待的装好笛头哨片，组装好管子以后就吹，发现怎么也没有声音。经过了半天的琢磨，请教博主，发现学习不能太急，按步骤来，再循序渐进。

萨克斯依赖固定在笛头上的哨片的震动来发声，因此要想吹奏的声音持久，稳定，就必须掌握好口型，还有吹奏时吐出的气流要恒定。这方面网上的教程有很多，推荐本人觉得不错的几位：金萨克斯张宏博老师的讲解，王老师的大众萨克斯系列；他们的教程和作品在西瓜视频，快手上都能搜到。

学习萨克斯的第一个阶段就是吹气，口型注意，然后练长音，不能着急练颤音，吐音这些花技巧。如果长音吹的不好，基础没打牢，后面的都白搭。

由于本人工作繁忙，工作日每天到家都10点多，为了不影响邻居休息，我都是在周末练习，目前进度是练长音。遇到的问题包括发出尖啸声、低音和高八度音吹不上去。请教了博主以后，发现吹这些音要注意把气沉下去，大致体验和慢呼气的原理类似，不要猛吹气，吹出的得是暖气才行。这个需要自己去悟，我目前在艰难修炼中。

乐理如何入门

如果以吹奏流行音乐为最终目的，可以用简谱上手，原因如下：

  • 简谱比五线谱更容易阅读，1-7分别代表do re mi fa so la xi 几个音阶。五线谱则是用圈圈在线行之间表示，没有前者易读。
  • 五线谱的表现形式对于我们来说比较生疏，符号很多，而吹奏流行乐曲可能不需要这么多的技巧符号。
  • 本人打算先从五线谱入手，慢慢来。简谱也熟悉。后面会主要靠简谱来学习。

乐理入门可以参考网上很多资料，买本萨克斯入门教材来学习，本人选用的是北京日报出版社的这本教材。

结篇

就到这里了，后面会陆续更新本人学习演奏技巧的其他环节。不管过程怎样，希望能坚持下去，成功入门自己心仪的乐器。

对于三年的工作小结

Veröffentlicht am 2019-08-11 | in reflection

[TOC]

正篇

技术和生活的现象。

技术方面

iOS

iOS开发没有专长，基础架构转业务开发以后一直处于业务驱动的状态。

其他技术栈

  • 大数据开发做了半年的项目中止，学了一些Spark, Hadoop，分布式相关的开发技巧，皮毛水平。

  • 自学Unity，跟完了一期课程，掌握了模糊的技巧和一些概念。

  • 音视频相关的内容掌握了一点皮毛。

生活

至今没有对象，单身狗。 不擅长社交，但是内心还是渴望社交的。

没有太多的业余爱好，间歇性失去对生活的信心，陷入抑郁。买房买车遥遥无期，旅游计划无限度搁置，每天有很多想法却不知道怎么实现。每周末打打游戏，结束以后都是万分的空虚。

思考

技术还是要走到业务驱动的路上来，自学自己啃的方式很难产生短期和长远收益，容易陷入弃坑或者一把乱抓的地步。

对所做的业务梳理和理解程度必须要达到一定的水平，这里急不了，也没法突击。跟需求的过程中要反复思考，敢于质疑，这方面还可以再加强一点。

培养一个业余爱好，可以学一门乐器，坚持两年看看。感情生活靠缘分，房产车产不强求。旅游可以等明年假期多了安排下。认清自己的定位，不属于一个技术宅男，就没有必要把大把经历耗在技术上，多去学习其他知识，做好准备面临随时都会来的裁员吧。

每天或者每周坚持刷题和总结，外企梦不能轻易放弃。

结尾

人生没有意义，我们的归宿都是尘土，在有限的时间里活出人生的宽度，不要着急，不要焦躁，也不要迷失方向。不沉迷于过去的一点成就和后悔曾经怎么样以后就会怎么样的无限死循环的假设推演去。

工作中的一点不顺利不要让它放大到生活，经常总结，避免经常性的重复犯错。奔三的年纪，承认自己不是强者，也不是弱者，只不过是平庸的又一代, 每天为了生机奔波。

__weak引发的一点思考

Veröffentlicht am 2017-06-04 | in iOS

背景介绍

最近在开发中遇到了一个有趣的问题：一个对象A，被对象B的属性a弱引用。在A的dealloc方法中，打印B->_a和B.a的值，发现前者正常打印，后者的结果是nil。这很好的勾起了我的好奇心和求知欲（对于__weak在 Runtime的实现理解不够透彻），邃手写demo，查阅资料；经过一番谷歌和LLVM的手册的搜索后，有了答案。

分析

起初以为编译器对weak属性的getter做了特殊处理，如果直接访问成员变量，则有值，不过这个假设在我修改了demo的代码以后就被推翻了：无论是通过getter访问还是直接访问成员变量，二者的值都是nil，只不过在控制台打印的结果二者会有不同。于是问题就变成了两个：weak属性在dealloc里的值为什么是nil？在NSLog等方法输出到控制台以后，以上两种取值方式的输出结果却不同？

对demo中的dealloc部分代码的汇编代码进行了分析，发现了一点猫腻（以下分别是__strong与__weak属性）在dealloc中的不同（省略了干扰代码）：

不难发现，对于__weak属性的访问，编译器调用了objc_loadWeakRetained函数，对于这个函数，LLVM文档第4.2节 给出了如下解释：

Reading occurs when performing a lvalue-to-rvalue conversion on an object lvalue.
For __weak objects, the current pointee is retained and then released at the end of the current full-expression. This must execute atomically with respect to assignments and to the final release of the pointee.
For all other objects, the lvalue is loaded with primitive semantics.

If object is registered as a __weak object, and the last value stored into object has not yet been deallocated or begun deallocation, retains that value and returns it. Otherwise returns null.

结合该方法源码上方的一段说明：

Once upon a time we eagerly cleared *referrer if we saw the referent was deallocating. This confuses code like NSPointerFunctions which tries to pre-flight the raw storage and assumes if the storage is zero then the weak system is done interfering.
That is false: the weak system is still going to check and clear the storage later.
This can cause objc_weak_error complaints and crashes.
So we now don’t touch the storage until deallocation completes.

结果

通过以上内容，我们知道了这个函数的作用：在__weak属性被访问时，为了确保其结果有值，或者为nil，不会出现__unsafe_unretained属性那种野指针的现象，编译器的处理为retain该对象，并加入自动释放池，在之后使用完毕后再释放。同时在dealloc方法中，此时对象已经处于deallocation的状态，objc_loadWeakRetained函数会返回nil。 第一个问题解决了！关于这个函数的实现，有兴趣的同学可以在objc的源码的objc-weak.mm中了解下，这里就不再对源码进行讲解。

对于问题二，目前还在调研中，我的猜测是编译器对getter和成员变量取值这两种形式做了特殊处理，使用了Code Generate技术。由于成员变量的释放与内存清理会在dealloc的末尾，也就是调用[super dealloc]中的 object_dispose函数中执行，因此编译器察觉到了这一点，为了方便Log，就保留了原值。 如果你有更好的解答，请在评论中回复我，非常感谢。

如何搭建React Native环境

React-Native开发的相关知识了解，前端开发的基本概念入门,包括框架介绍，控件布局，样式，网络请求，更多资源参考

为什么要做原生组件的封装呢？React Native为我们提供了很多现成组件，比如ScrollView，Navigator，TextInput；但是在实际开发中，我们许多时候中会遇到React Native没有为我们备好的组件的需求，而原生项目中已经有在用的控件了，后者往往来自于第三方库或者App自带。如果能把这些控件用JS封装好，并应用于React Native项目中，在性能上要优于后者，体验也更流畅；而React Native为我们提供了这样的入口，这是非常好的。

新的开始

Veröffentlicht am 2016-10-23 | in reflection

序言

转眼工作也有一年了，一直想写点东西，算作对自己在过去时间里的总结，但是由于各种事情的耽搁，加上自己的一点轻微拖延症的影响，这个想法始终未能实现。最近这段时间，这个想法在脑海中愈发的强烈，最终我决定去注册个人域名，搭建自己的博客网站，把内心的想法付诸行动，有了这个开端，未来的写作之路相信会逐渐铺延展开，我也会定期在上面分享生活和工作中的体会心得。

回顾

毕业离开学校到现在的这一年里，我陆续跳槽两次，经历一次转型，从当初死磕大数据到最后选择移动开发，这期间的过程充满了艰辛和无奈，可以说，自14年底读研以来，我就走了一年多的弯路，错过和浪费了多次校招的机会，在深圳广州之间来往了多次，期间电话面试了多个知名互联网公司，跑到北京后也没有松懈，在收到某通信公司的offer以后，我理应松口气，但是当时一点也高兴不起来。

因为我知道，这不是我理想的工作，第一是工作的方向不喜欢，如果最后是这种结局，我还不如本科毕业的时候就找工作去28呢，可能待遇和环境还比这个工作好些。第二是我认为我应该去更好的互联网公司，做更上手的工作，拿比现在更好的待遇，而不是以先锻炼几年再跳槽来当借口安慰自己一时的胆怯和不自信。

我开始反思自己过去的种种应聘失败的原因：自己的失败一方面是时机把握不对，在面一家电商公司的大数据开发时候已经进入了二面，但是因为对未来的发展摇摆不定，放弃了继续面试的想法；另一方面是准备不足，过于迷信运气，不知道当务之急是什么，海投简历，乱闯一气，最后却竹篮打水一场空。还有一点，我投递大数据开发是严重的从众心理，忽略了自己的长处，以自己的弱项去生搬硬套，最后只能自吞苦果。

痛定思痛以后，我决定放手一搏，选择目前应届生竞争相对较小的移动开发作为我进入互联网的切入点，为了有充分的时间去自学，我选择了离职。在这里我要感谢自己的爸妈，在关键时候给予了自己充分的支持和肯定，也感谢自己的表妹，把mac借给了我去学习。方向确定，决心坚定，就没有不成功的道理，尽管学习的那段时间很辛苦，早起晚归，但是我知道那是在和时间赛跑，在与命运相搏，此时若松懈放弃，就再没回头的路。几个月后，我如愿以偿拿到了一家创业公司的offer，得到了比之前的公司高近一倍的薪水，工作也走上了正轨。

总结

人不可能重复地犯同一个错误，不怕屡次跌倒，就怕不去思考和总结自己失败的原因；像个无头苍蝇一样在乱撞，只会跌的更惨。在工作中，我也时常把从这段经历中得到的教训作为前车之鉴来警戒自己。要懂得把握机遇，要懂得知足，对自己的能力和未来要做到心中有数；永远要知道目前正在做的是什么,这些对今后尽可能地少走弯路，减少犯相同错误有着极大的帮助。
